Paragon · Catalog Match Take-Home

DESIGN PROCESS

How I thought through the problem, the decisions I made along the way, and how I used AI to build it — from first principles to shipped prototype.

01 — Problem framing

THE PROBLEM

Before writing a single line of code, I needed to understand what made this genuinely hard — not just "match two strings" hard.

Customer request
"SHCS 7/16 x 2-1/2 alloy"
"same washers as last time"
"M8 bolt zinc 500 pcs"
→

Catalog description
"HX HD SCR 7/16-14X2-1/2 A GR5 ZN"
"FLT WSHR 1/2 SS 18-8"
"HX HD SCR M8-1.25X20 A SS316"
Key insight
Both sides are free text. This is not a query-against-structured-database problem. Structure must be derived from the descriptions, not assumed to exist. That changes everything about the approach.

02 — Before writing code

QUESTIONS FIRST

I wrote out the questions that needed answers before any design decision could be made. These drove everything that followed.

On matching
How do we reliably match one piece of free-form text against another?
How do we handle abbreviations and shorthand that differ between query and catalog?
What happens when the description doesn't fully specify all the relevant attributes?
What happens when it references attributes that aren't in the catalog at all?
On confidence
What is the right algorithm for a confidence score in this domain?
How should the score behave under vague input, abbreviations, or missing attributes?
What does a high-confidence match look like versus a coin-flip between two candidates?
Should low confidence block a recommendation or just surface a warning?
On system design
What does clean separation between the data layer, matching logic, and UI actually look like in practice?
Should there be a minimum description length before a match is attempted?
Lexical vs. semantic — what are the real tradeoffs for this specific domain?

03 — Thinking through failure modes

EDGE CASES

I identified these in two passes — first the obvious ones, then a second round that came from actually reading the catalog data carefully.

Pass 1 — Obvious from the brief
Abbreviations & shorthand
Customer writes "SHCS 7/16 x 2-1/2" — catalog has "SKT HD CAP SCR 7/16-14X2-1/2". Normalization must go both ways.
Near-identical SKUs
Multiple items that differ only by coating, length, or standard. The right answer depends on which variant the customer actually needs.
Vague / sparse descriptions
"Washer" or "bolt" could match dozens of SKUs. Score should visibly collapse — not pretend certainty.
Attributes not in the catalog
Customer asks for grade 8, tolerance class, or surface roughness. No SKU can verify this — surface the gap rather than ignore it.
Confidence score sensitivity
Same numeric score means different things for rich vs. sparse queries. Score separation from #2 matters as much as the raw score.
Pass 2 — Found in the data
Quantity contamination
"Need 500 M8 hex nuts" — strip leading counts before matching or BM25 will penalize the query.
Multi-product queries
"M8 nuts and washers" — detect and warn rather than silently returning one confused best match.
Referential queries
"Same as last time" — pure order-history lookup when a customer is selected. No matching needed.
Metric vs. imperial collision
M8 and 5/16-18 should never cross-match. These are completely separate namespaces. Hard filter, not just a penalty.
Misspellings
"sockt head cap scrw" — Levenshtein fallback when BM25 retrieval is weak. Budget-capped to avoid pathological cost.
Mixed units in one query
"M8 screw 1 inch long" — metric thread spec + imperial length. Parse thread and length independently.

05 — Before writing the matcher

DESIGN PRINCIPLES

These were the constraints I set before touching the scorer. If a design decision violated one of these, I went back and reconsidered.

Principle 01
Precision over throughput
Never pretend certainty when score separation is weak. A wrong auto-match is worse than routing to review.
Principle 02
Human review is part of the system
Review Required is not a failure mode — it's a product decision. Some queries should escalate to a human by design.
Principle 03
Explainability built in
Every result shows score components, a rationale, and what the system could not verify. No black boxes for operators.
Principle 04
History is a prior, not a override
Customer order history fills silence. It never overrides an explicit attribute in the current query. The user's words win.
Principle 05
Normalize before matching
Token alignment between query and catalog index must be guaranteed. One shared normalization module — no drift.
Principle 06
Metric and imperial never cross
These are separate namespaces. Hard pre-filter, not a soft penalty. An M8 and a 5/16 should never be compared.
Principle 07
Clean layer separation
build.js generates data. normalization.js normalizes — used by both build and browser. matcher.js scores — no DOM. app.js renders — no scoring. Four files, four jobs, no overlap.

06 — The calls that shaped the system

KEY DECISIONS

Each of these was a real fork in the road. Here's the question I faced and the answer I landed on.

Decision A When the query says "zinc" but the customer's history is full of stainless — which wins?
Query wins, always. History is a prior for filling gaps, not a signal to override what the user explicitly typed. If someone writes "zinc coating," that's the intent — history suggesting stainless gets suppressed on that attribute entirely.
Decision B How much weight should a customer's order history carry when they've only bought 2 things?
None. Below 3 orders, I treat the customer as a new account and apply zero history prior. Two data points isn't a pattern — applying it would add noise, not signal. The threshold is enforced at both build time and runtime.
Decision C Should low-confidence queries be blocked or should results always be returned?
Always return top 3, but route to Review Required. Blocking produces nothing useful. Returning results with a clear Review Required decision and an honest confidence score lets the operator see the candidates and make the call. Pretending low confidence = no answer is the wrong tradeoff.
Decision D What happens when the user asks for grade 8 or a specific tolerance and nothing in the catalog has that field?
Match on what we can, surface what we can't verify. The matcher still returns the best candidates on the attributes it does understand. An "Unverified request details" panel explicitly calls out that grade, tolerance, and heat treatment are not encoded in the catalog — so the operator knows the match is incomplete, not wrong.
Decision E Should inactive / discontinued SKUs be excluded from results?
No — include them, but label clearly. Excluding them silently could hide the best technical match and leave the operator confused about why nothing came back. Better to surface the match with a visible "⚠ Inactive SKU" tag and let the operator decide what to do with it.

09 — What changed after the first build

ENGINEERING REFINEMENTS

These are the things I discovered when the system was running on real queries that I hadn't anticipated from the design alone.

Bug fix
Lag screw vs. hex cap screw — pattern ordering collision
The catalog description "HX HD LAG SCR" was matching as a hex cap screw because the "HX HD" pattern fired before the lag screw pattern. Fixed by moving all lag screw patterns to the top of the PRODUCT_PATTERNS array — priority matters in this kind of regex dispatch.
Bug fix
BM25 token mismatch — "1/2 rod 6 foot" returning wrong result
Two sub-problems: "foot" wasn't expanding to "ft" so BM25 never matched the catalog token "6ft". And "1/2" bare diameter never matched catalog token "1/2-13". Fixed by adding feet/foot expansion to ABBREV_PAIRS and indexing bare diameter separately at build time.
Fix
Attribute penalty too weak — flat washer ranking above hex nut
The original 0.5x type-conflict multiplier wasn't strong enough — thread-spec overlap was overcoming it. Replaced the multiplier with a hard disqualifier that returns 0.02 when product types explicitly conflict. Solved the problem cleanly.
Architecture
Normalization extracted to shared module
Originally normalization lived in both build.js and matcher.js as duplicated code. This created a class of bugs where a new abbreviation added in one place but not the other would cause query/catalog token mismatches. Extracted to normalization.js — one source of truth, used by both.
Feature
Decision routing added — Auto-Match vs. Review Required
The original spec returned confidence labels but didn't make an explicit routing decision. Added the decision object (route, label, message, reasons) because I realized the UI needed to communicate clearly whether the system was confident enough to proceed or whether a human should review. This also made the score breakdown more meaningful.

10 — Honest accounting

TRADEOFFS AND LIMITS

These are the deliberate simplifications I made for the take-home, and what a production version would look like instead.

This prototype — deliberately simplified
⚡
BM25 + attribute rerank only. No embedding tier. Fast, explainable, and easy to defend in a live conversation.
⚡
Heuristic multi-product detection. Works on "and/plus/comma" separators and plural product types. Won't catch everything.
⚡
Mock review handoff. "Select SKU" moves to a mock panel — no real ERP, inventory, or order-submission backend.
⚡
Static data layer. data.js is regenerated from CSVs. No live inventory sync or real-time catalog updates.
⚡
Ambiguous bare metric lengths (e.g. "M8 x 1" without unit) are still inherently ambiguous — the system surfaces this rather than guessing.
Production — what I'd add next
🔜
Hybrid retrieval: BM25 + embedding similarity with RRF fusion. Better at synonyms and semantic gaps that normalization misses.
🔜
Full intent parsing for multi-product queries. NLP-based product type extraction rather than heuristic keyword counting.
🔜
Real order handoff — ERP / inventory API integration. The routing decision the system makes would trigger an actual workflow.
🔜
Feedback loop. Reviewed edge cases become labeled training data. The system improves from every human correction.
🔜
Customer history from a live database rather than a pre-baked CSV. Real-time recency decay and profile updates.
